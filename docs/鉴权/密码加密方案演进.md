# 密码加密方案演进

> **日期**：2026-07-18
> **标签**：密码学, HMAC, bcrypt, 前端安全

---

## 演进过程

### V0：无加密

```
浏览器 → 明文 "admin123" → POST → 服务端 → bcrypt → 数据库
```

**问题**：如果错误日志记录了 request body，明文密码会暴露在日志中。

### V1：SHA-256

```
浏览器 → SHA256("admin123") → POST → 服务端 → bcrypt → 数据库
```

**优点**：日志中永远看不到明文。

**问题**：
- 无盐 — 彩虹表可预计算 `"admin123" → "8d969aef0e..."`，批量反查
- SHA-256 有长度扩展攻击风险

### V2：HMAC-SHA256（当前方案）✅

```
浏览器 → HMAC-SHA256(key=username, message=password)
       → POST → 服务端 → bcrypt → 数据库
```

```typescript
// packages/shared/src/crypto.ts
const key = await crypto.subtle.importKey('raw', encoder.encode(username), { name: 'HMAC', hash: 'SHA-256' })
const sig = await crypto.subtle.sign('HMAC', key, encoder.encode(password))
```

---

## 为什么 HMAC 更好

| 攻击类型 | SHA-256 | HMAC-SHA256(username) |
|------|:--:|:--:|
| 彩虹表 | ❌ 同一密码 → 同一哈希，可批量反查 | ✅ `alice:admin123` ≠ `bob:admin123` |
| 长度扩展攻击 | ❌ SHA-256 固有弱点 | ✅ HMAC 免疫 |
| 密码长度熵 | ❌ 64 字符固定输出 | ❌ 相同（都是 64 字符） |

**核心理由**：以 `username` 为 HMAC key，攻击者需要为每个用户名单独建彩虹表，成本指数上升。

---

## 为什么不直接传明文 + HTTPS？

这是业界大多数产品（GitHub、Google、Notion）的做法。但要避免一个坑：**服务端日志记录了 request body**。

```typescript
// ❌ 危险 — 如果 Nginx/中间件记录完整请求体
logger.info('Login request:', req.body)  // { username: "alice", password: "myp@ss" }
// 这会写入磁盘 → 被日志分析系统采集 → 明文密码遍布多个系统
```

客户端哈希的价值就是：**即使日志泄露，攻击者也只能拿到 HMAC 哈希，无法用彩虹表批量反查明文密码。**

---

## 权衡

| 不做客户端哈希 | 做客户端哈希 (HMAC) |
|:--|:--|
| 简单 | 复杂度 +1 |
| 依赖 TLS 加密通道 | 多一层纵深防御 |
| 日志泄露 → 明文密码 | 日志泄露 → HMAC 哈希（有盐） |
| bcrypt 接收任意长度密码 | bcrypt 接收固定 64 字符 HMAC |

**结论**：客户端哈希不是"必须做"，而是"纵深防御"。如果你要求"永远不让明文密码离开浏览器"，就值得这一层。
